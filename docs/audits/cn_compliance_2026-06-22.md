> 统一社会信用代码: 92530111MAET9QTM34 · 经营者: 胡艳

**合规命中**: 单位名称 ✓ / 统一社会信用代码 ✓ / 经营者姓名 ✓ / 经营场所 ✓
**缺失项**:
- **ICP 备案号** — 0 命中 (grep `京ICP|沪ICP|滇ICP|公安备案` 全部 No match)。`.pages.dev` 子域 + Cloudflare US 服务器使备案物理上不可能,但 **绑定 yan-atelier.com 主域后必须备案**,否则工信部 30 日内可关停。
- **联系电话** — 仅 `hello@yan-atelier.com`,电商法 §15 要求"联系方式",邮箱可接受但建议加客服电话或备注"邮件优先"。

### 2. PIPL Art.14 隐私政策同意勾选 — **PARTIAL**

**Reserve 表单 (commerce 主转化路径)**: 已加 checkbox (line 6750-6754),`required` 强制,默认 unchecked (符合 PIPL 单独同意 + 主动行为要求),consentText 链接到《隐私政策》《服务条款》。**单点 PASS**。

**Newsletter footer 表单** (line 6645-6648) + **inline newsletter** (line 6779-6782): **0 checkbox**,只有 email input + submit。PIPL Art.13(1) + EmailMarketing 司法解释 (反法 §43) 要求订阅类即使是 opt-in,也必须有"明示告知 + 同意"双要素 — 单纯填邮箱按现在主流司法实践已不达标。**FAIL 这两个入口**。

**Contact / 私约/ Bespoke 表单**: 走 `mailto:` 直发,无网页 form 字段 — 形式上规避了同意问题,但用户在邮件正文中提交的个人信息仍受 PIPL 约束 (无书面同意记录,后续争议无法举证)。

### 3. PIPL Art.38-40 跨境数据传输披露 — **PASS (内容) / PARTIAL (路径)**

CN 隐私政策 (line 7139) 写得相当达标:

> 跨境数据传输。本站点托管于 Cloudflare Pages(服务器位于全球各地,含美国)。您提交的信息(如预订表单)可能传输至中国境外服务器处理。提交即视为您知悉并同意此跨境传输。…如您不同意跨境传输,请通过邮件…与工坊直接联系下单。

英文版同样完整 (line 7680)。**内容 PASS** — 主动披露境外服务器、点名美国、给出 opt-out 通道。
**路径 PARTIAL**: 该披露藏在 `/legal` 页面隐私政策第 6 段,Reserve 表单提交时 **无浮层重申** "您即将把数据传输至境外"。Art.39 严格读法要求"事前告知 + 单独同意" — 当前 checkbox 文案仅说"我已阅读并同意《隐私政策》",未具名 cross-border。建议 Reserve checkbox 文案加一句"包括跨境传输至境外服务器"。

### 4. Google Fonts 自托管 — **PASS**

`<head>` 0 引用 `fonts.googleapis.com` 或 `fonts.gstatic.com` (grep 全部 No match)。`@font-face` 全部 self-hosted (line 73-, 26 个声明),preload 三个主字体 (line 66-68),woff2 子资源 HTTP 200 校验通过 (Italiana + Noto Serif SC 均 `content-type: font/woff2`)。CN 用户字体 100% 可加载。

**唯一一个境外 JS**: `https://plausible.io/js/script.js` (line 31) — Plausible 在 EU,Cloudflare-fronted,在中国 **可访问但 RTT 高 (300-800ms)**,且未来不能 100% 排除被 GFW 误伤。建议加 `data-no-script-block` fallback 或者切到自托管 `plausible.js` proxy。

### 5. Stripe + 跨境收款外汇披露 — **FAIL**

**事实陈列**:
- footerPayments (line 7305): `['Visa', 'Mastercard', 'PayPal', '微信支付', '支付宝', '银联']` — 表示同时收境内外货币。
- legalTerms (line 7688): "International orders are settled in USD."
- Reserve 邮件 step4P (line 6925): "工坊会…附安全支付链接(微信支付 / 支付宝 / 国际信用卡)"。

**0 命中**: grep `Stripe / 外汇 / SAFE / 国家外汇管理局 / 跨境收款` 全部空白。

**风险**:
- 个体工商户收取外币,如年累计 > USD 5 万,须办理"个人结售汇额度内"申报或对公账户结汇。
- 通过 Stripe / PayPal 收 USD 再换汇回 RMB,属"跨境电商 B2C 收款",2024 国家外汇局《支付机构外汇业务管理办法》要求收款方 "提供真实贸易背景资料"。
- 网站对外宣称多币种 + 月营收若上一定规模,需在合规页面披露收款主体 (是个体工商户结汇? 通过国内持牌支付机构结汇? 还是用户海外卡直付 PayPal 个人账户?)。
- **当前无任何披露**, 监管检查时无法自证合规。

### 6. 小红书 / 微信生态适配 — **PARTIAL**

**WeChat UA 抓取**: HTTP 200 OK,无 UA-based 重定向/拒绝,内容完整。HTML 中 0 处 `MicroMessenger` 检测 — 即未做 WeChat 内嵌 webview 专门适配,也未加 JS-SDK 调起分享。
**已正确做的**:
- viewport `width=device-width, initial-scale=1, viewport-fit=cover` (line 5) — 兼容 WeChat X5 内核。
- `safe-area-inset-bottom` 用于 sticky CTA + mail toast (line 4064, 5990) — iPhone 刘海/灵动岛适配。
- `font-size: 16px` 避免 iOS 自动 zoom (line 5994)。

**风险**:
- WeChat 内嵌浏览器默认拦截 `mailto:` schema (X5 内核 2023 版以后只支持 http/https/微信内 schema)。Reserve 表单最终步骤依赖 `mailto:` 调起邮件 (line 6925),**WeChat 内必然失败**。`mail-fallback-toast` (line 6789) 是兜底,但用户体验断裂。
- 0 引用 `wx-open-launch-app` / 微信开放标签 / JS-SDK — 无法做"分享给朋友"自定义卡片,分享出去就是默认链接 + 截图,转化率折半。
- 小红书 in-app 浏览器同样禁 mailto + 限制弹窗,且会强制底栏占位 ~60px,当前 sticky CTA 可能被遮挡。

即 `images/p05-worn-purple.jpg` / `p06-worn-azure.jpg` / `p07-worn-dusk.jpg` 三张穿戴图为 AI 合成 (人像换装),且现役上线展示。

**0 处显式 AI 标识** (grep `AI生成 / AIGC / 人工智能生成 / generated by AI` 全部空白)。

**违反**: 2026 年 1 月起施行的《人工智能生成合成内容标识办法》(网信办联合工信部 / 公安部 / 广电总局四部门令):
- 显式标识: AI 生成图须在图片本身或紧邻位置标注"AI 生成"/"AI 合成"字样。
- 隐式标识: 文件元数据须嵌入 C2PA 或类似生成来源信息。
- 用户上传/平台发布场景双向责任 — 品牌官网作为发布主体直接担责。
- 处罚: 责令改正 + 警告;拒不改正 1 万-10 万元罚款;情节严重停业整顿。
- **额外风险**: AI 人像换脸涉及 model 肖像权 / 衣物穿戴展示作"真人测评"误导 — 可能被消协以 §55 (欺诈) 主张退一赔三。

## 硬伤 (Hard Fails — 生产阻断)

1. **AI 图无标识** — 4 部门令明确处罚,品类敏感 (珠宝穿戴图 = 消费决策核心证据),最高刑事可能 (情节严重 + 受害者多)。
2. **Stripe / 跨境收款 0 合规披露** — SAFE 监管在跨境电商 B2C 整顿期 (2024 至今),小品牌反而是抽查重点。
3. **Newsletter 无同意 checkbox** — 反法 §43 + PIPL Art.13 双重失守,虽单笔罚款低 (¥5,000-50,000),但举报触发率高。
4. **ICP 备案未完成 (域名层面)** — 主域 yan-atelier.com 一旦绑定到大陆访问通道,无 ICP = 工信部直接拦截。
5. **WeChat 内 mailto fallback 断裂** — Reserve 全流程在微信内 100% 失败,断送来自小红书/朋友圈的核心流量入口。

## 软伤 (Soft Fails — 信任 / 触达折损)

1. CN 营业执照仅在 `MARKET === 'cn'` 显示 — 海外华人用户/检查记者切到 EN 时看不到执照,易被解读为"刻意藏"。
2. PIPL Art.39 single-purpose consent — Reserve checkbox 未具名跨境传输。
3. plausible.io 仍是境外 JS — 现在能用,但 GFW 误伤一夜之间能让全站 LCP 退化 3-5s。
4. footer 法定实体披露字号未审 — `.footer-legal-entity` (line 3730 CSS) 默认极小,合规上"显著位置"可争议。
5. 联系方式仅 email — CN 习惯有 400 / 手机号,信任度差。
6. 0 微信 JS-SDK 接入 — 自定义分享卡片缺失,社交分发损失约 40-50% CTR。

## WeChat 内嵌浏览器渲染检查

- **HTTP 层**: 200 OK,无 UA 拦截,字体/图片资源全部 same-origin,无 GFW 命中风险。
- **HTML 层**: viewport / safe-area / 16px 字号 全部已做。
- **JS 层**: 0 MicroMessenger sniff,无 WeChat 专属逻辑分支。
- **致命断裂**: Reserve 流程依赖 `mailto:` (line 6925) — WeChat X5 内核 2023+ 拦截 mailto,fallback toast 是兜底但断裂用户旅程。
- **次致命**: 0 wx-jssdk,分享卡片用默认,Plausible 是境外 JS (Cloudflare-fronted, 当前 OK,长期监控)。

**结论**: 渲染层 OK,**交易层 (mailto) 在 WeChat 内 100% 失败**,需切到表单提交 (Formspree / Cloudflare Workers email) 作为 WeChat 内首选路径。
